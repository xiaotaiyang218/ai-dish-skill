# OpenClaw 核心用例手测清单

本文档用于把 `dish-health-recommender` 安装到 OpenClaw 或其他 Agent 后，按自然语言对话方式验证 6 个核心用例。用例不依赖结构化 JSON 输入，目标是模拟真实用户在手机 Web 页面或 Agent App 里的问答过程。

## 测试前准备

- 建议新开一个对话，避免历史健康信息干扰结果。
- 默认用中文自然语言提问，不需要显式要求 JSON。
- 如果平台支持图片输入，可以上传对应菜图或菜单图；如果当前平台不能识别图片，应接受它降级为“需要确认”，并继续用文字补充菜名或候选菜名。
- 外部 API 不是必需条件。弱网或离线时，系统应基于本地规则、菜名归一、常见配方和健康约束给出核心判断。
- 输出不要求逐字一致，但必须命中“推荐等级、主要理由、待确认项、健康风险优先级”。
- 所有结果仅作为饮食参考，不应表述为医疗诊断或治疗建议。

## 推荐输出格式

OpenClaw 实测时，建议检查回答是否包含以下信息：

- 客观信息：识别到的菜名、可能食材、常见做法、风险标签；有标准份量数据时补充热量、蛋白质、脂肪、碳水、糖、钠等营养估算。
- 结论：`推荐`、`谨慎`、`不推荐` 或 `需要确认`。
- 原因：为什么这些食材、做法或用户约束会影响结论。
- 建议：换菜、少量尝味、少油少盐、补充菜名或食材等可执行动作。
- 需要确认：配方、过敏源交叉接触、份量、图片置信度等。

## 用例 1：鸡蛋过敏 + 番茄炒蛋

### 验证目标

验证过敏规则优先级最高。即使番茄炒蛋是常见家常菜，只要用户鸡蛋过敏，就应直接给出 `不推荐`。

### 对话脚本

用户：

```text
我鸡蛋过敏，番茄炒蛋还能吃吗？
```

### 期望结论

`不推荐`

### 期望回答要点

- 能识别用户约束是 `鸡蛋过敏`。
- 能把 `番茄炒蛋` 识别为通常含有 `鸡蛋` 的标准菜。
- 说明过敏风险优先级高于普通营养评价。
- 建议除非能确认完全不含鸡蛋或蛋制品，否则不要点。

### 验收点

- 必须出现：`不推荐`。
- 必须提到：鸡蛋或蛋制品风险。
- 不能只说“少吃一点”或“谨慎吃”，因为过敏不是普通减脂/控糖问题。

### 覆盖能力

- 健康约束驱动推荐。
- 标准菜名与常见配方匹配。
- 过敏硬规则覆盖普通推荐。

## 用例 2：减脂/控糖 + 上海红烧肉

### 验证目标

验证系统能把多重健康目标和标准配方风险组合起来解释，而不是只按菜名给泛泛建议。

### 对话脚本

用户：

```text
我最近在减脂，也要控糖，上海红烧肉可以吃吗？
```

### 期望结论

`谨慎`

### 期望回答要点

- 能识别用户目标：`减脂`、`控糖`。
- 能识别上海红烧肉常见风险：五花肉、高脂、糖、酱油、红烧汁、钠。
- 说明不一定绝对不能吃，但不适合作为主菜或大量食用。
- 如果输出营养量化，热量单位必须写清楚为 `kcal` 或 `千卡`。
- 如果给出标准估算，可参考：1 份约 `540 kcal`，主要来自五花肉、糖、酱油和烹调油吸收。

### 验收点

- 必须出现：`谨慎`。
- 必须同时解释 `减脂` 和 `控糖` 两个约束。
- 建议应包含少量尝味、控制份量、避免汤汁/浓油赤酱等可执行建议。
- 如果给精确数值，必须说明是标准份量估算，不能装作实测克重。

### 覆盖能力

- 健康约束驱动推荐。
- 多源知识增强与营养量化。
- 上海本帮菜标准配方风险解释。

## 用例 3：海鲜过敏 + 鱼香肉丝

### 验证目标

验证菜名歧义消除能力。`鱼香肉丝` 名字里有“鱼”，但通常是风味名，不应直接误判为一定含鱼；同时因为用户海鲜过敏，需要保守确认。

### 对话脚本

用户：

```text
我海鲜过敏，鱼香肉丝能吃吗？名字里有鱼。
```

### 期望结论

`需要确认`

### 期望回答要点

- 解释 `鱼香` 通常是风味名，不等于一定含鱼或海鲜。
- 不能简单因为有“鱼”字就直接判断为海鲜菜。
- 因为用户海鲜过敏，仍需要确认餐厅实际配方、酱料和后厨交叉接触。
- 建议询问店员或查看配料表，确认无鱼露、海鲜调味料、共用锅具污染等风险。

### 验收点

- 必须出现：`需要确认`。
- 必须提到：`鱼香` 是菜名/风味歧义，不等于一定有鱼。
- 必须保留过敏风险提醒，不能直接说“可以放心吃”。

### 覆盖能力

- 菜名归一与歧义消除。
- 不确定性显式降级。
- 过敏场景下的保守策略。

## 用例 4：低盐 + 招牌小炒

### 验证目标

验证系统面对模糊菜名时不硬猜。`招牌小炒` 不是标准菜名，无法可靠判断主要食材、酱料和盐分。

### 对话脚本

用户：

```text
我今天要低盐，菜单上有个招牌小炒，可以点吗？
```

### 期望结论

`需要确认`

### 期望回答要点

- 说明 `招牌小炒` 不是标准菜名，信息不足。
- 需要用户补充主要食材、烹饪方式、酱汁或菜单图片。
- 低盐场景下，应特别确认是否用了酱油、豆瓣酱、腌制肉类、咸菜、重口味复合调料。
- 可以给出临时建议：让店家少盐少酱、酱汁另放，或换清蒸/白灼/清炒类菜。

### 验收点

- 必须出现：`需要确认`。
- 必须说明不能可靠知道配方和用盐量。
- 不能编造具体食材和精确钠含量。

### 覆盖能力

- 模糊菜名保守处理。
- 信息充分性检查。
- 不确定性降级。

## 用例 5：菜单/菜肴图片候选识别与筛选

### 验证目标

验证多模态链路：图片或菜单识别先得到候选菜名，再结合用户健康约束逐菜筛选。这个用例也验证弱网或无图像能力时的降级策略。

### 方式 A：平台支持图片输入

用户先上传一张上海菜菜单图或菜肴候选图，然后发送：

```text
我海鲜过敏，还想低盐减脂。请从这张图里帮我筛选能吃的菜，并告诉我为什么。
```

### 方式 B：平台不支持图片输入或图片识别不稳定

用户直接发送候选菜名：

```text
这张上海菜菜单里识别出了油爆虾、水晶虾仁、腌笃鲜、草头圈子。我海鲜过敏，还想低盐减脂，帮我筛一下，并说明每道菜的风险。
```

### 期望结论

逐菜筛选，而不是只返回一个笼统结论。

### 期望回答要点

- `油爆虾`：命中虾类/海鲜过敏，`不推荐`。
- `水晶虾仁`：命中虾类/海鲜过敏，`不推荐`。
- `腌笃鲜`：通常含咸肉、笋、汤汁，低盐和减脂场景下 `谨慎`，尤其少喝汤、确认少盐。
- `草头圈子`：通常含猪大肠、红烧汁和油脂，减脂/低盐场景下 `谨慎`。
- 如果图片识别低置信或多菜同屏，应说明“图片候选需要人工确认”，不能把候选当成完全确定事实。
- 如果输出推荐排序，应优先排除过敏硬风险，再比较低盐/减脂风险。

### 验收点

- 必须区分每一道菜，而不是只说“都不建议”。
- 必须把虾类菜判为 `不推荐`。
- 必须说明图片/OCR 候选存在不确定性时需要确认。
- 如果平台无法看图，应明确降级并要求用户提供菜名或 OCR 文本。

### 覆盖能力

- 多模态识别与候选召回。
- 健康约束逐菜筛选。
- 低置信图片显式降级。
- 档口/候选信息可作为推荐解释的一部分。

## 用例 6：人工反馈纠错 + 标准量化闭环

### 验证目标

验证系统不是一次性识别工具。它应能先处理识别不确定，再接受用户纠正菜名，并基于纠正后的标准菜重新输出推荐和热量估算。

### 对话脚本

第一轮用户：

```text
我在减脂，这张菜图你先判断一下能不能吃。它看起来像猪蹄煲。
```

如果平台支持图片输入，可以同时上传草头圈子或类似红烧菜图片；如果不支持图片输入，只用文字也可以。

第一轮期望回答：

- 如果无法确认菜名，应输出 `需要确认`。
- 说明“猪蹄煲”或图片候选不可靠，不能直接给确定结论。
- 请求用户确认真实菜名或主要食材。

第二轮用户：

```text
不是猪蹄煲，实际是草头圈子。
```

第二轮期望结论：

`谨慎`

### 期望回答要点

- 明确接受用户反馈：把候选从 `猪蹄煲` 纠正为 `草头圈子`。
- 重新判断草头圈子的风险：猪大肠、红烧汁、烹调油、钠和脂肪风险。
- 结合减脂目标给出 `谨慎`，建议少量、少油、少汁，不作为主菜。
- 如果输出热量量化，可参考标准估算：1 份约 `430 kcal`。
- 量化解释可包含：猪大肠、草头、烹调油、酱油、糖、红烧汁等分项来源。

### 验收点

- 必须体现反馈纠错：不能继续坚持“猪蹄煲”。
- 必须在纠正后重新推荐。
- 必须出现：`草头圈子`、`谨慎`、`减脂` 相关解释。
- 如果给热量，必须写清 `kcal` 或 `千卡`，并说明是标准份量估算。

### 覆盖能力

- 图片/候选不确定性降级。
- 人工反馈闭环。
- 标准菜谱营养量化。
- 用户纠错后重新计算推荐。

## 常见失败信号

- 对过敏场景只输出“少吃一点”或“谨慎”，没有给出明确避开建议。
- 把 `鱼香肉丝` 直接当作含鱼菜，或反过来在海鲜过敏场景下直接说“放心吃”。
- 对 `招牌小炒` 编造具体食材、钠含量或热量。
- 图片识别低置信时仍给确定菜名和确定推荐，不提示需要确认。
- 用户纠正菜名后，系统没有更新判断对象。
- 输出热量时不写单位，或在缺少份量/食材克数时给出看似精确的数值但不说明估算依据。
